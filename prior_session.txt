
Goal
Build an idiomatic Terraform provider for Jamf Protect backed by a GraphQL API. The project has gone through schema improvements, resource implementation, data source implementation, and is now implementing 8 high-impact provider improvements to make the resources work in a more modern Terraform-defined way. After that, the user wants a Python script to convert Jamf Protect custom analytic YAML files from the github.com/jamf/jamfprotect repo into Terraform HCL.

Discoveries
1. Plan resource analyticSets input uses uuid field (not analyticSet) in the PlanAnalyticSetInput
2. Plan field values: signaturesFeedConfig.mode = "blocking" (not ON/OFF), commsConfig.protocol = "mqtt" (lowercase), analyticSets[].type = "Report"/"Prevent", logLevel = "ERROR"
3. Info sync valid attrs: arch, hostName, kernelVersion, memorySize, modelName, osMajor, osMinor, osPatch, osString, serial, insights
4. Analytic inputType valid values: GPFSEvent, GPDownloadEvent, GPProcessEvent, GPScreenshotEvent, GPKeylogRegisterEvent, GPClickEvent, GPMRTEvent, GPUSBEvent, GPGatekeeperEvent
5. Analytic analyticActions.parameters is a JSON-encoded string in the API (e.g. "{\"id\":\"smartgroup\"}")
6. Analytic actions parameters field is AWSJSON! (non-null) — when no parameters are provided, the provider must send "{}" (empty JSON object), not omit the field. On read-back, "{}" must be mapped to null in Terraform state to avoid "inconsistent result after apply" errors.
7. PreventList tags are NOT returned by the API's getPreventList query — the provider uses a manual preservation hack to keep them in state
8. Reference providers studied: terraform-provider-jamfpro and terraform-provider-jamfplatform — both use stringvalidator.OneOf() for enums, RequiresReplace() for immutable fields, Optional+Computed+Default pattern, nested blocks, and snake_case field naming with case conversion
9. terraform-plugin-framework-validators v0.19.0 and terraform-plugin-framework-timeouts v0.7.0 are dependencies
10. The GraphQL client respects context cancellation — uses http.NewRequestWithContext(ctx, ...) on both token and query paths
11. Alert config structure: The alertConfig JSON has structure { "data": { "<eventType>": { "attrs": [...], "related": [...] } } } with exactly 14 event types
12. Event type name mapping (snake_case → camelCase): binary→binary, click_event→clickEvent, download_event→downloadEvent, file→file, fs_event→fsEvent, group→group, proc_event→procEvent, process→process, screenshot_event→screenshotEvent, usb_event→usbEvent, user→user, gk_event→gkEvent, keylog_register_event→keylogRegisterEvent, mrt_event→mrtEvent
13. USBControlSet API: Polymorphic rules field with VendorRule (vendors list), SerialRule (serials list), ProductRule (vendor+product pairs). defaultMountAction enum: ReadOnly, ReadWrite, Prevented. No $RBAC_Plan directive needed.
14. TelemetryV2 API: No getTelemetryV2 query was captured from browser DevTools — the provider constructs one following the same pattern.
15. ID field discrepancy: Some resources use id (Plan, ActionConfig, PreventList, TelemetryV2, USBControlSet) while others use uuid (Analytic, UnifiedLoggingFilter). The get queries reflect this (getPlan(id: $id) vs getAnalytic(uuid: $uuid)).
16. Pagination pattern: All list queries use token-based pagination with input: {next: $nextToken, order: {direction, field}, pageSize: 100} and return {items{...}, pageInfo{next, total}}. Exception: listAnalytics takes no input parameters at all.
17. List query field coverage: listActionConfigs only returns basic fields (id, name, description, created, updated) — NOT the full alertConfig.
18. listUnifiedLoggingFilters requires a non-null $filter: UnifiedLoggingFiltersFilterInput! parameter.
19. Import pattern issue: All 7 resources do a full API read inside ImportState(), then Terraform automatically calls Read() again — resulting in a double API call on every import. Modern best practice is resource.ImportStatePassthroughID() to just set the ID and let Read() handle the rest.
20. Missing UseStateForUnknown on created and hash: These computed fields show "(known after apply)" noise on every plan even though they never change (or only change when config changes).
21. description has Optional+Computed but no default: Causes "(known after apply)" when omitted.
22. No ErrNotFound sentinel error: isNotFoundError() relies on fragile string matching ("not found"/"not_found") against ErrGraphQL.
23. No User-Agent header in HTTP requests to the Jamf Protect API.
24. Analytic level has no range validator — accepts any Int64 value.
25. Prevent list missing updated attribute (all other resources have it).
26. Jamf Protect custom analytic YAML schema maps closely to jamfprotect_analytic resource. Key YAML fields: name, uuid, longDescription→description, inputType→input_type, filter, level, severity, tags, categories, snapshotFiles→snapshot_files, actions[].name→analytic_actions[].name, context→context. Extra YAML-only fields: shortDescription, label, version, remediation, MitreCategories. All ~80 YAML files use level: 0, actions: [{name: Log}], and only 4 of 9 input types.

Key implementation details for each improvement:
Item 1 (ImportStatePassthroughID): Replace the custom ImportState methods in all 7 *_resource.go files. Current pattern (example from action_config_resource.go lines 332-354) does a full API query + apiToState; replace with single line resource.ImportStatePassthroughID(ctx, path.Root("id"), req, resp). The Read() method already handles full state population. Special case: prevent_list_resource.go — import currently sets data.Tags = types.ListNull(types.StringType) because tags aren't returned by the API. With passthrough ID, Read() will need to handle this (it already does via stateTags preservation, but on import there won't be prior state tags — need to verify this still works).
Item 2 (UseStateForUnknown on created): Add PlanModifiers: []planmodifier.String{stringplanmodifier.UseStateForUnknown()} to the created attribute in all 7 resource schemas. Exact locations:
- action_config_resource.go line 155-158
- analytic_resource.go line 149-152
- plan_resource.go line 168-171
- prevent_list_resource.go line 80-83
- telemetry_v2_resource.go line 85-88
- unified_logging_filter_resource.go line 81-84
- usb_control_set_resource.go line 125-128
Item 3 (UseStateForUnknown on hash): Add to hash attribute on action_config_resource.go line 60-63 and plan_resource.go line 52-54.
Item 4 (description default): Add Default: stringdefault.StaticString("") to the description attribute in all 7 resources. Will need to add the import "github.com/hashicorp/terraform-plugin-framework/resource/schema/stringdefault" where not already present.
Item 5 (ValidateConfig): Need to identify cross-attribute validation rules. Example: USB control set rules should have vendors set when type is VendorRule, serials when SerialRule, products when ProductRule.
Item 6 (ErrNotFound): Add ErrNotFound = errors.New("jamfprotect: resource not found") to internal/graphql/client.go. In the Query method, check GraphQL error messages for "not found"/"not_found" and wrap with ErrNotFound instead of ErrGraphQL. Update isNotFoundError() in helpers.go to use errors.Is(err, graphql.ErrNotFound). Update tests in helpers_test.go and client_test.go.
Item 7 (User-Agent): Add a userAgent field to the Client struct in internal/graphql/client.go. Set it in NewClient() (need to accept version param or set via method). Add req.Header.Set("User-Agent", c.userAgent) to both the authenticate() and Query() methods. The version string is available in main.go as version var, passed to provider.New(version).
Item 8 (level validator): Add int64validator.Between(0, 10) (or appropriate range) to analytic_resource.go line 80-83. Will need import "github.com/hashicorp/terraform-plugin-framework-validators/int64validator" and int64validator import.
